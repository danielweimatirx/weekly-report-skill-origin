## 事项 3：工作代码与产出统一进 matrixorigin 仓库

### 背景

当前 skill 通过共享 token 采集**公司 org 范围内**的 PR / Issue 数据（`org:matrixorigin` 及其他公司 org）。这意味着：

- ✅ 公司 org 内的 repo → 全覆盖
- ❌ **放在个人账号下的 repo → 完全采集不到**（机器账号 token 无权读）

导致：
1. 产品团队做的**原型**（常用个人账号 repo）→ 不在周报里
2. 市场团队做的**官网/博客/素材**（如果在个人号）→ 不在周报里
3. 研发的 **PoC / demo**（临时放个人号）→ 不在周报里
4. 客户相关的**售前 demo / 行业研究**（放个人号）→ 不在周报里

员工付出的大量工作就这样"看不见"。不只影响周报，还影响代码资产沉淀（员工离职仓库带走或被删）。

### 推进目标

**凡是可以通过代码和 GitHub Issue 产生的工作产出，统一放到 `matrixorigin`（或 `matrixone-cloud` 等公司 org）的仓库里**，包括但不限于：

- PoC（proof of concept）
- 产品原型（如 PM 做的交互原型、设计稿驱动的代码）
- Demo（客户演示 demo、售前 demo）
- 市场素材（可以版本化管理的：博客源文件、白皮书、宣传视频脚本等）
- 行业调研 / 客户档案（结构化数据 + 代码组织）
- 内部工具 / 脚本
- 培训资料 / 文档

### 负责方

**赵晨阳**（研发 VP，AI 平台 leader，有决策权和推动力）+ **张旭**（tech lead，熟悉各仓库和员工实际工作情况）**联合推动**。

### 建议执行步骤

1. **启动阶段**（赵晨阳 + 张旭 1 周内完成）
   - 盘点各团队目前哪些重要工作放在个人号（可以让团队 leader 上报一份清单）
   - 评估是否需要新建专门的 org（如 `matrixorigin-prototypes`、`matrixorigin-demo`、`matrixorigin-labs`）还是直接放入现有 org
   - 制定**迁移优先级**：先迁当前活跃 + 客户相关的，历史低频项目慢慢迁

2. **规范发布阶段**
   - 发布内部规范文档：**新建 repo 默认建在公司 org**，不建在个人号
   - 特殊情况（纯个人学习/开源贡献）才允许放个人号
   - 文档说明为什么（代码资产、周报可见性、离职交接等）

3. **存量迁移阶段**（1-2 个月滚动推进）
   - 按团队逐个盘点 + 迁移（`gh repo transfer` 命令可保留历史）
   - 公司 org 内按团队建 team 管理可见性（避免敏感 demo 公开）

4. **工具侧配合**（skill 侧）
   - 不用改 skill 本身
   - 迁移完成后，原本看不见的工作自然纳入周报
   - 如果要分类统计（"哪些是 PoC"），后续可以约定 repo 命名前缀或 topic 标签

### 为什么是赵晨阳 + 张旭

- **赵晨阳**：VP 级别，跨 AI 平台 / 平台开发 / 应用开发三个子组，有决策权推动新建 org、发布规范
- **张旭**：tech lead 级别，熟悉各 repo 和各成员实际工作，能具体识别"哪些是工作、哪些是纯个人"

建议产出物：
- 一份《矩阵起源代码资产归属规范》文档（1-2 页）
- 一份存量迁移清单（谁的哪个 repo → 迁到哪里）
- 每周进展跟踪

## 三件事的关系与时间线

```
事项 1（别名补齐，HR 主导）──┐
事项 2（Leader 补齐，企微管理员）─┤──→ 完成后 skill 全员识别无死角，周报质量跃升
事项 3（代码归 org，赵晨阳+张旭）─┘

时间线建议：
  Week 1：事项 1 + 2（纯数据治理，1 周内可完成）
  Week 2-3：事项 3 启动阶段（赵晨阳 + 张旭对齐盘点）
  Month 2：事项 3 存量滚动迁移完成
```
